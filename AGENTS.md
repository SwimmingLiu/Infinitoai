# AGENTS.md

## Purpose

本文件用于约束本仓库内所有自动化代理/协作者在**完成编码、修改、重构、修复、打包、文档更新等任何会修改文件的工作后**的标准执行流程。

> 只要改过文件，就不要直接宣称“完成”，而应优先按下面的 workflow 执行。

---

## Required Workflow After Any File Change

当你在本项目中进行了任何文件修改后，**建议默认按照以下 workflow 执行**：

### 1. 完整测试

在提交 PR 之前，尽可能完成以下测试：

- 单元测试（unit tests）
- 端到端测试（e2e tests）
- 像素级/视觉回归测试（pixel / visual regression tests）

执行原则：

- 优先运行与本次改动直接相关的测试。
- 如果仓库已具备完整测试能力，尽量执行全量测试。
- 如果某类测试当前仓库尚未配置，应在结果里明确说明缺失项，而不是假装已完成。
- 如果测试失败，先修复问题，再重新执行测试。

### 1.1 推荐测试方案（当前项目默认建议）

结合本项目的形态（Chrome 扩展、Side Panel、多页面跳转、OAuth 流程、邮箱/验证码链路、UI 状态变化），推荐默认采用以下测试方案：

#### 端到端测试（E2E）

**首选：Playwright**

推荐原因：

- 更适合真实浏览器驱动和真实用户流程模拟
- 更适合处理 Chrome 扩展、Side Panel、tab 切换、弹窗、新页面跳转
- 更适合覆盖 OAuth、验证码、邮箱轮询、回调地址写回等复杂链路
- 支持网络拦截、trace、录像、失败回放，便于排查问题

默认建议：

- 优先使用 `@playwright/test`
- 对关键流程建立稳定的端到端用例
- 对失败场景、回退场景、重试场景补充回归用例

#### 像素级 / 视觉回归测试

**首选：Playwright 原生截图断言**

推荐方式：

- `expect(page).toHaveScreenshot()`
- `expect(locator).toHaveScreenshot()`

推荐原因：

- 接入成本最低
- 可直接集成到现有 E2E 流程中
- 适合 Side Panel、Toast、日志区、步骤状态区等 UI 稳定区域做像素级回归
- 本地和 CI 都容易落地

视觉测试重点建议覆盖：

- Side Panel 主界面
- 关键步骤状态变化
- Toast / 错误提示 / 成功提示
- 日志区域滚动与布局
- 不同配置条件下的主要 UI 形态

#### 真实场景测试

对于“真实页面 + 真实邮箱 / 验证码链路 + 真实 OAuth 流程”的高仿真测试，仍然推荐使用 **Playwright** 作为主执行框架。

建议分层执行：

- 日常 PR：运行核心回归 E2E + 关键视觉回归测试
- 夜间构建 / 手动触发 / 发版前：运行真实场景链路测试

原因：

- 真实场景测试通常更慢、更脆弱、更依赖外部环境
- 不适合每次提交都全量执行
- 更适合作为 nightly、pre-release 或手动验证流程

#### 邮件链路 / 测试环境建议

为了提高稳定性，建议优先使用可控测试环境：

- 本地或测试环境邮箱服务，例如 `Inbucket`
- 专用测试账号、测试域名、测试面板
- 可重复使用的测试数据与固定回放场景

如果必须接入真实外部服务，应在测试说明中明确：

- 环境依赖
- 凭据来源
- 运行前置条件
- 不稳定因素与重试策略

#### 视觉协作增强（可选）

如果后续需要在 GitHub PR 中更直观地审查视觉差异，可以在 Playwright 原生截图方案稳定后，再考虑接入：

- `Percy`

默认建议：

- 第一阶段先使用 Playwright 自带 screenshot diff
- 只有当团队确实需要 PR 级视觉审查平台时，再引入 Percy

#### 不优先推荐的方案

当前项目默认**不优先推荐 Cypress 作为主 E2E 方案**，原因包括：

- 对 Chrome 扩展、Side Panel、多标签页、真实浏览器级行为的适配通常不如 Playwright 自然
- 对本项目这种跨页面、多步骤、强依赖浏览器上下文的流程，Playwright 更合适

#### 默认测试技术栈结论

如果仓库没有特别说明，则默认推荐测试技术栈为：

- 单元测试：沿用仓库现有单元测试框架
- 端到端测试：`Playwright`
- 像素级 / 视觉回归测试：`Playwright screenshot assertions`
- 真实邮件链路测试环境：`Inbucket` 或其他可控测试邮箱环境
- PR 视觉审查增强：按需选配 `Percy`



### 2. 提交 GitHub PR

测试通过后，提交 GitHub Pull Request。

要求：

- PR 标题准确描述本次修改目的。
- PR 描述应包含：变更内容、测试范围、风险点、是否涉及 UI/行为变化。
- 如有必要，附上截图、录屏或测试结果摘要。

### 3. 本地 Code Review

PR 创建后，进行 **本地 Code Review** 即可，不再依赖 `@copilot` 或 GitHub Copilot 自动 review。

本地 Review 处理方式：

- 基于本次变更的 diff、相关文件和测试结果进行逐项检查
- 重点关注：功能正确性、回归风险、边界条件、异常处理、日志、可维护性
- 如有 UI / 行为变化，结合截图、录屏、Playwright 回放或本地页面实际表现一起审查
- 对 review 中发现的问题直接继续修改，并重新进入测试 / PR 更新流程

要求：

- 不要在本地 review 未完成前直接视为最终完成。
- 不要忽略本地 review 中发现的关键问题。
- 如果当前环境无法完成充分的本地 review，必须明确说明阻塞点和未覆盖风险。

### 4. 根据 Review 继续修改代码

如果本地 review 或测试结果提出问题，需要继续修改代码。

**一旦再次修改了文件，就重新回到上述 workflow 起点**，继续执行：

1. 完整测试
2. 提交/更新 GitHub PR
3. 完成本地 Code Review
4. 根据反馈继续修改

### 5. 循环直到满足合并条件

重复上述流程，直到同时满足以下条件：

- 所有应执行的测试均通过
- 没有未处理的本地 review 问题
- PR 状态满足合并条件

只有在以上条件满足后，才可以合并 PR。

## Packaging Requirement

在完成代码修改并准备交付时，还应尽量进行打包，产出**可运行文件或安装包**。

要求：

- 如果仓库已有打包脚本，直接使用既有脚本产出构建结果。
- 如果仓库缺少明确打包脚本，但本次任务需要交付可运行产物，应补充合适的打包方式。
- 打包完成后，应说明产物位置、运行方式、适用平台，以及已知限制。
- 如果当前项目天然属于“源码型项目”且尚不具备安装包能力，也要在交付说明中明确当前实际可交付形式。

## Default Behavior Expectations

对于本仓库内的任何 agent / contributor，默认遵循以下行为：

- 只要修改了文件，就不要跳过测试、PR、review、回改、再测试这一闭环。
- 不要把“代码写完”当作“任务完成”。
- 不要在没有测试依据的情况下声称“已修复”或“可用”。
- 不要在未经过 review 闭环时直接结束协作。
- 如果受限于环境、权限、CI、网络或缺失脚本而无法完成某一步，必须明确说明阻塞点。

---

## Short Version

**只要修改了文件，推荐默认流程：**

1. 完整测试：unit / e2e / pixel
2. 提交 GitHub PR
3. 完成本地 Code Review
4. 根据反馈改代码
5. 重复以上流程，直到测试通过且本地 review 完成，再合并 PR
6. 打包代码，输出可运行文件或安装包
